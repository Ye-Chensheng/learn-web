# GitHub 版本控制与协作

## 1. 核心概念与工作流 (Core Concepts)

在深入界面功能之前，必须理解 Git 的核心数据流向。这是所有操作的基石。

### 1.1 三大工作区

Git 的操作本质上是文件在三个区域间的流转：

1. **工作区 (Working Directory)**：本地电脑上的文件夹，用户在此编辑代码。
2. **暂存区 (Staging Area)**：一个临时区域，用于存放即将提交的文件快照（相当于“购物车”）。
3. **本地仓库 (Local Repository)**：安全存储所有版本历史记录的地方（相当于“本地存档”）。

### 1.2 关键指令解析

- **`git add .`**：将工作区的修改添加到暂存区。

- `git commit -m "message"`

  ：将暂存区的内容封装成一个版本（Commit），永久保存到本地仓库。

  - *注意*：此时代码仅在本地，云端不可见。

- **`git push`**：将本地仓库的新版本推送到远程仓库（GitHub），实现云端同步。

## 2. GitHub 界面功能详解 (Interface & Modules)

基于 GitHub 网页端主要模块的深度解析。

### 2.1 Code (代码/仓库主页)

仓库的“门面”，展示文件结构与核心信息。

- **分支切换器 (Branch Switcher)**：查看不同分支（如 `main`, `feature/user-page`）下的文件状态。
- **Clone/Download**：获取仓库地址（HTTPS/SSH），用于将云端代码下载到本地。
- **README.md**：项目的说明书，通常包含项目简介、安装步骤和使用文档。

### 2.2 Pull Requests (PR / 拉取请求) —— **核心重点**

PR 是 GitHub 团队协作的核心机制，它不是一个单纯的“合并”动作，而是一个**“请求代码审查与合并”的过程**。

- **定义**：开发者在分支完成开发后，请求管理员将代码合并到主分支。

- 关键要素

  ：

  - **Base Branch**：目标分支（通常是 `main` 或 `develop`）。
  - **Compare Branch**：源分支（你开发的功能分支，如 `feature/user-page`）。
  - **Diff (差异对比)**：系统自动高亮显示变更内容（绿色代表新增，红色代表删除），供审查者阅读。

- 协作流程

  ：

  1. **发起 (Open)**：提交代码并填写变更描述。
  2. **审查 (Review)**：团队成员在 PR 页面逐行阅读代码，提出修改意见（Comment）。
  3. **迭代 (Update)**：开发者在本地修改并再次 Push，PR 会自动更新，无需重新发起。
  4. **合并 (Merge)**：审查通过后，点击 Merge 按钮将代码并入主分支。

### 2.3 Projects (项目看板)

用于项目管理的敏捷开发工具，类似 Trello 或 Jira。

- **功能**：将 Issue（问题）和 PR 转化为可视化的任务卡片。

- 视图 (Views)

  ：

  - **Kanban (看板)**：分为 `Todo` (待办), `In Progress` (进行中), `Done` (已完成) 等列。
  - **Table (表格)**：类似 Excel 的清单视图。

- **作用**：管理者分配任务，开发者追踪进度，确保项目按计划推进。

### 2.4 Insights (洞察与统计)

仓库的数据分析中心。

- **Network**：以图形化方式展示分支的分叉与合并历史，用于追溯版本演变。
- **Contributors**：统计贡献者活跃度（提交次数、代码行数）。
- **Dependency Graph**：分析项目依赖的第三方库。

### 2.5 Settings (设置与安全)

- Security Overview

  ：

  - **Secret Scanning**：检测代码中是否意外泄露了密钥、Token 等敏感信息。
  - **Dependabot**：自动检测依赖库是否存在安全漏洞，并提示升级。

- **Branch Protection Rules**：设置分支保护规则（详见第 4 节）。

## 3. 分支管理策略 (Branching Strategy)

在企业级开发中，严禁直接在主分支操作。通常采用 **Git Flow** 或其变体进行管理。

### 3.1 分支类型定义

| 分支名称          | 作用                                                   | 权限                                | 生命周期       |
| :---------------- | :----------------------------------------------------- | :---------------------------------- | :------------- |
| **main / master** | **生产环境**。存放随时可发布的稳定代码。               | 仅允许 Merge，严禁直接 Push。       | 长期存在       |
| **develop**       | **开发主轴**。所有开发者的代码汇聚于此，进行集成测试。 | 允许组长 Merge，通常禁止直接 Push。 | 长期存在       |
| **feature/xxx**   | **功能分支**。用于开发新功能（如 `feature/login`）。   | 开发者自由读写。                    | 功能上线后删除 |
| **hotfix/xxx**    | **热修复分支**。用于紧急修复线上 Bug。                 | 开发者自由读写。                    | 修复后删除     |
| **release/xxx**   | **预发布分支**。用于版本发布前的最终测试。             | 冻结新功能，仅修 Bug。              | 发布后删除     |

### 3.2 常用命名规范

- 功能开发：`feature/user-page`, `feat/login-module`
- Bug 修复：`fix/nav-bar-error`
- 文档修改：`docs/update-readme`

## 4. 企业级开发 vs 个人开发 (Corporate vs. Individual)

从“写代码”到“工程化”的跨越，主要体现在流程规范和自动化上。

### 4.1 核心区别对照表

| 维度         | 个人/学习阶段                    | 互联网公司/工程化阶段                                        |
| :----------- | :------------------------------- | :----------------------------------------------------------- |
| **代码提交** | 直接 `git push` 到 `main` 分支。 | **禁止**直接 Push 到 `main`。必须使用分支开发 + PR 合并。    |
| **代码质量** | 自己觉得没问题就行。             | **Code Review (代码审查)**：必须由 1-2 名同事审核通过（Approve）才能合并。 |
| **合并权限** | 随意 Merge。                     | **分支保护 (Branch Protection)**：GitHub 强制限制，未满足条件（如测试未通过）无法点击合并按钮。 |
| **测试流程** | 手动运行代码看效果。             | **CI/CD (持续集成/部署)**：提交 PR 后，服务器自动运行自动化测试脚本，全绿灯才允许通过。 |
| **任务管理** | 想到什么写什么。                 | **基于 Issue/Ticket**：每个 PR 必须关联一个任务单号（如 Jira ID）。 |

### 4.2 真实工作流示例

1. **领任务**：在 Projects 看板领取任务“开发用户详情页”。
2. **切分支**：本地终端执行 `git checkout -b feature/user-detail`。
3. **开发提交**：编写代码 -> `git add` -> `git commit`。
4. **发起 PR**：Push 到远程，在 GitHub 发起 Pull Request，指定 Reviewer（审查人）。
5. **机器检查**：GitHub Actions 自动运行测试（Lint检查、单元测试），若失败需立即修复。
6. **人工审查**：同事提出“变量命名不规范”，你在本地修改后再次 Push。
7. **合并上线**：同事 Approve，CI 测试通过 -> 点击 Merge -> 代码进入 `develop` 或 `main`。

## 5. 拓展知识与常见问题 (Extensions & FAQ)

### 5.1 什么是 .gitignore？

- **定义**：一个配置文件，告诉 Git **忽略**哪些文件。
- **场景**：系统自动生成的文件（如 `.DS_Store`）、依赖包文件夹（如 `node_modules`）、敏感配置文件（如包含密码的 `.env`）。这些文件不应上传到仓库。

### 5.2 遇到“冲突 (Conflict)”怎么办？

- **原因**：两个人同时修改了同一个文件的同一行代码，Git 无法判断保留谁的。

- 解决

  ：

  1. 在本地执行 `git pull` 拉取最新代码。
  2. 打开代码编辑器，会看到 `<<<<<< HEAD` 和 `>>>>>>` 标记。
  3. 人工判断，手动删除不需要的代码，保留正确的代码。
  4. 重新 `add`, `commit`, `push`。

### 5.3 如何安全地合并 feature 到 main？(操作指引)

基于你当前的进度，标准操作如下：

1. **不要在终端直接 Merge**（除非你是单人开发且非常熟练）。

2. 在 GitHub 网页端发起 PR

   ：

   - 点击 `Compare & pull request`。
   - 确认标题清晰（如 `feat: add self-learning HTML/CSS notes`）。
   - 点击 `Create pull request`。

3. 执行合并

   ：

   - 检查页面显示的 `This branch has no conflicts with the base branch`。
   - 点击 `Merge pull request` -> `Confirm merge`。

4. 本地同步

   ：

   - 回到终端，切换回主分支：`git checkout main`
   - 拉取云端最新合并后的代码：`git pull`
   - *此时，你的本地 main 分支才真正拥有了 feature 分支的内容。*
